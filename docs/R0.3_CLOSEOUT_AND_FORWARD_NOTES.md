## Active Security Note: `qs@6.15.1`

OSV Scanner reported:

- advisory: `GHSA-q8mj-m7cp-5q26`
- CVE: `CVE-2026-8723`
- severity: moderate
- affected versions: `>=6.11.1 <6.15.2`
- installed version: `6.15.1`
- fixed version: `6.15.2`

The package is an indirect dependency used by Express and `body-parser` in the
API server workspace. The vulnerable behavior requires application code to
call `qs.stringify` with both:

```js
{
  arrayFormat: 'comma',
  encodeValuesOnly: true
}
```

and provide an array containing `null` or `undefined`.

No direct use of `qs.stringify` or that option combination was found in Inner
Atlas source during R0.3 closeout. The dependency and vulnerability were
already present on `origin/main`; R0.3 did not introduce them.

Decision:

- not a blocker for the completed R0.3 audio-only refactor
- still an active dependency risk
- update to a dependency graph resolving `qs >= 6.15.2` before beta/release
- handle in a separate dependency/security commit with build, typecheck,
  API smoke tests, and OSV rescan

Do not silence or ignore the advisory without either upgrading or documenting
a deliberate time-bounded exception.

## Active Architecture Note: Layer Builders Receive the Full Engine

This concern remains active.

Current construction uses calls such as:

```js
buildRain(this);
buildAnalog(this);
```

Each layer builder therefore receives the complete `AudioEngine` object rather
than a narrow dependency contract. This preserved behavior and reduced wiring
risk during R0.3, but it allows builders to reach engine state, private fields,
timer handles, gain maps, and node trackers.

Decision:

- not a correctness bug
- not an R0.3 blocker
- do not refactor it opportunistically in R0.4 unless Resonance makes a narrow
  interface necessary
- reassess before or during R0.5 Scene Architecture

R0.5 is the natural decision point because scenes will parameterize layer
mixes and may require independently testable layer construction.

A future narrow contract could expose only:

```text
AudioContext
master and reverb buses
final-gain registration
source/LFO tracking
layer-specific scheduler hooks
read-only state or an audible-state predicate
```

Do not replace the engine object with a large generic "context" object that
merely renames the same coupling.

## R0.4 Notes

R0.4 replaces or hides the continuous Pulse layer with event-based Resonance.

Before implementation:

- preserve compatibility with stored mixes containing `pulse`
- decide whether `pulse` is ignored, migrated, or retained internally
- design Resonance as a one-shot/event source, not a continuously looping layer
- define timer and scheduler ownership explicitly
- avoid adding another engine property such as `_resonanceTimer` without first
  considering a small scheduler/timer registry
- preserve the transition mutex and audible-state guards
- retain human listening QA as an acceptance gate

R0.4 may move the `tape` compatibility alias out of `buildAnalog` and explicitly
null `_dropletBuf` and `_analogCrackleBuf` during disposal, but only if those
changes remain small and behavior-preserving.

## R0.5 Notes

R0.5 introduces Scene Architecture and is the required checkpoint for the
builder boundary.

Before implementation:

- define the Scene data contract independently from Web Audio node objects
- keep stored scene/mix data serializable and versionable
- preserve older sessions with no `scene` field
- keep layer builders independent from product labels and UI state
- decide whether layer builders return owned resources and cleanup callbacks
- define timer/scheduler ownership before adding scene-driven events
- add focused tests for mix application and compatibility aliases

At R0.5 planning, explicitly classify the full-engine builder coupling as one
of:

1. refactor now because scenes require isolation
2. retain temporarily with documented constraints
3. replace with a tested narrow builder contract

Do not let this decision remain implicit.

## Other Deferred R0.3 Notes

These remain non-blocking:

- `stereoMotion.js` contains graph-construction helpers; consider renaming it to
  `graphHelpers.js` or `audioHelpers.js` if R0.4 introduces real stereo-motion
  effects
- the `tape` alias is currently registered by the analog layer builder rather
  than the engine compatibility boundary
- `_dropletBuf` and `_analogCrackleBuf` are not explicitly nulled in
  `dispose()`, although they are replaced on graph rebuild
- droplet and crackle timers remain individual engine properties
- named exports from `audio/audioEngine.js` widen its public surface; prefer
  importing state definitions from `audioStateMachine.js`

These items should not be bundled into unrelated feature work.

## Merge Discipline

For R0.4 and R0.5:

- keep product implementation commits separate from local tool inventories
- keep process/router changes separate unless the milestone explicitly depends
  on them
- do not commit generated review artifacts that duplicate Git history
- do not commit incomplete templates as authoritative reports
- run build, typecheck, lint, Gitleaks, and OSV when dependencies change
- require Claude review before merge for architecture-sensitive audio changes
- record human listening results before claiming audio acceptance
